# 全局设置页与自动清理（审计日志 / 备份）

日期：2026-07-01
状态：待确认

## 一、目标

1. 新增 Admin UI「全局设置」菜单页。
2. 支持启用「自动删除审计日志」，按设置天数自动删除过期审计记录。
3. 支持启用「自动删除备份文件」，按设置天数自动删除过期备份文件。
4. 把审计日志页底部「手动清理」按钮迁移到全局设置页（保留下拉，档位改为 10/20/30/50）。

- 配置唯一来源：`config.json` → `DatabasesConfig`（`src/McpDbTools.Server/Configuration/DatabaseConfig.cs:103`）。全局字段（如 `defaultMaxConcurrency`）挂在 `DatabasesConfig` 顶层。

- 手动清理 API：`POST /admin/api/audit-logs/cleanup`（`Program.cs:143-159`），body `{name: "<days>"}`，前端调用在 `logs.js:476-501`，UI 在分页栏（`logs.js:398-419`），档位固定 30/60/90。
- 备份管理：`AdminConfigService.ListBackups`（`AdminConfigService.cs:355`）枚举 `backups/config.*.json`，备份项 `time` 取文件 `LastWriteTimeUtc`。无自动删除。
- SPA 导航数据驱动：`nav.js` 的 `NAV_ITEMS`（`nav.js:6-11`）；视图注册到 `window.adminViews[id]`（见 `projects.js`、`keywords.js`、`logs.js`、`backups.js`）；shell 按 hash 路由（`shell.js`）。
- 带「保存」的视图（projects/keywords）模式：`GET /admin/api/config` 拉全量 → 缓存到 `state.config` → 表单编辑 → `PUT /admin/api/config` 全量回写，保存时必须携带 projects 等其它字段否则后端全量替换会丢数据。
- 配置热重载：`ConfigStore` 监听文件变更去抖重载（`ConfigStore.cs:79-99`）。Admin 保存经 `WriteConfigAtomicallyAsync`（`AdminConfigService.cs:320`）落盘后会触发热重载。
- 宿主：`--admin-only` 模式只启 ASP.NET Core Minimal API（`Program.cs:46`）；MCP 模式不启 Admin。当前 DI 注册见 `ConfigureBusinessServices`（`Program.cs:201-218`）。
- 测试惯例：`AdminBackupAndConnectionTests` 用临时目录 + `ConfigStore`/`AdminConfigService` 直接构造测试（`src/McpDbTools.Tests/AdminBackupAndConnectionTests.cs`）；`AuditLoggerTests` 测试审计写入/清理。

### 去重排查（Dedupe Ticket）
- Intent signature：`自动周期删除 audit_log 与备份文件 + 全局设置存储`
- Queries：`grep IHostedService|BackgroundService|Timer|Cleanup|Settings`（后端）；`grep settings|autoCleanup`（前端）
- Top matches：无 IHostedService/BackgroundService；无 settings 页面；无自动清理逻辑。仅 `ConfigStore._debounceTimer`（热重载去抖，非业务调度）、`AuditLogger.DeleteOlderThan`（手动清理，可复用）。
- Decision：新增 `MaintenanceHostedService`（IHostedService + 周期 Timer）实现自动清理；复用 `AuditLogger.DeleteOlderThan`；复用 `AdminConfigService` 备份枚举能力（新增删除过期方法）。配置字段新增到 `DatabasesConfig`，前端新增 settings.js 视图。
- Rationale：无重复实现；调度用 ASP.NET Core 标准 IHostedService 模式，符合 .NET 惯例；清理复用已有删除逻辑。

## 三、设计方案

### 3.1 配置存储（config.json）

在 `DatabasesConfig` 新增一个 `maintenance` 节点（运维清理设置）：

```jsonc
{
  "maintenance": {
    "auditLogAutoCleanup": false,
    "auditLogRetentionDays": 30,
    "backupAutoCleanup": false,
    "backupRetentionDays": 30
  },
  "databases": { ... }
}
```

新增模型 `MaintenanceConfig`（新文件 `Configuration/MaintenanceConfig.cs`）：

```csharp
public sealed class MaintenanceConfig
{
    [JsonPropertyName("auditLogAutoCleanup")]
    public bool AuditLogAutoCleanup { get; init; }

    [JsonPropertyName("auditLogRetentionDays")]
    public int AuditLogRetentionDays { get; init; } = 30;

    [JsonPropertyName("backupAutoCleanup")]
    public bool BackupAutoCleanup { get; init; }

    [JsonPropertyName("backupRetentionDays")]
    public int BackupRetentionDays { get; init; } = 30;
}
```

`DatabasesConfig` 新增：
```csharp
[JsonPropertyName("maintenance")]
public MaintenanceConfig? Maintenance { get; init; }
```
为 null 时按默认值（全 false / 30 天）处理，向后兼容旧 config.json。

### 3.2 自动清理后台服务

新增 `src/McpDbTools.Server/Maintenance/MaintenanceHostedService.cs`：
- 实现 `IHostedService` + `IDisposable`。
- 构造注入 `ConfigStore`、`AuditLogger`、`AdminConfigService`、`ILogger`。
- 启动时启动一个 `Timer`，周期 **1 小时** 触发一次 `DoCleanupAsync`，启动后 **30 秒**首次执行（避免与启动竞争）。
- `DoCleanupAsync`：
  1. 读 `ConfigStore.Current.Maintenance`（null 视为默认，全不启用）。
  2. 若 `AuditLogAutoCleanup` 且 `AuditLogRetentionDays > 0`：调 `AuditLogger.DeleteOlderThan(retentionDays)`，日志记录删除条数。
  3. 若 `BackupAutoCleanup` 且 `BackupRetentionDays > 0`：枚举备份，删除 `LastWriteTimeUtc < now - retentionDays` 的文件，日志记录删除数。
  4. 任何异常 catch + LogError，不抛出（后台服务不能挂）。

`AdminConfigService` 新增方法：
```csharp
public int DeleteBackupsOlderThan(int days)
```
逻辑：遍历 `ListBackups()` 同源文件，`LastWriteTimeUtc < cutoff` 的删除，返回删除数。复用 `IsSafeBackupName` 安全校验路径。

DI 注册：在 `ConfigureBusinessServices` 注册 `services.AddHostedService<MaintenanceHostedService>()`。该服务在 Admin 模式和混合模式生效；纯 MCP 模式（不启 ASP.NET Core generic host）不运行——但纯 MCP 模式审计仍在写入，清理延后到下次 Admin/混合模式运行时执行，符合「本机工具」定位（见风险节）。

### 3.3 Admin API

新增接口：
- `GET /admin/api/maintenance` → 返回当前 `MaintenanceConfig`（null 视为默认）。
- `PUT /admin/api/maintenance` → 保存 maintenance 配置。

DTO（加到 `Admin/AdminModels.cs` 或新建 `Admin/AdminMaintenanceModels.cs`）：
```csharp
public sealed class MaintenanceSettingsResponse
{
    public bool AuditLogAutoCleanup { get; init; }
    public int AuditLogRetentionDays { get; init; }
    public bool BackupAutoCleanup { get; init; }
    public int BackupRetentionDays { get; init; }
}
```

保存策略选择 **独立接口**（不走 `PUT /admin/api/config` 全量替换）：
- 理由：settings 页与 projects/keywords 页是不同视图，各自独立保存避免互相全量覆盖风险；maintenance 是独立 JSON 节点，单独读改写最安全。
- 实现：新增 `AdminConfigService.SaveMaintenanceAsync(MaintenanceSettingsRequest, ct)`：
  1. 读当前 `DatabasesConfig`（`ConfigStore.Current`）。
  2. 把 `maintenance` 节点替换为新值（其它字段不动）。
  3. 用与 `WriteConfigAtomicallyAsync` 相同的原子写回（复用，或抽公共方法）。
  - ⚠ 复用：为避免重复，把 `WriteConfigAtomicallyAsync` 现有的「写临时文件→校验→备份→替换」流程抽成可复用，或直接调用已有逻辑。本方案倾向**最小改动**：`SaveMaintenanceAsync` 内部序列化完整 `DatabasesConfig`（含新 maintenance）后走原子写。需保证不丢其它字段——直接基于 `ConfigStore.Current` 深拷贝替换 maintenance 节点即可。

后端校验：`retentionDays` 必须 `> 0`（与 `DeleteOlderThan` 一致），否则 400。

手动清理 API `POST /admin/api/audit-logs/cleanup` **保留不变**（前端只是改 UI 位置，接口照旧）。

## 六、风险与边界

1. **纯 MCP 模式不运行自动清理**：`MaintenanceHostedService` 依赖 ASP.NET Core generic host，纯 stdio MCP 模式（`RunMcpAsync`）用 `Host.CreateApplicationBuilder` 也注册了 `ConfigureBusinessServices`——需确认 `AddHostedService` 在 MCP host 下也会启动（Microsoft.Extensions.Hosting generic host 支持 IHostedService）。**实现时验证**：若 MCP host 也启动该服务，则审计自动清理在 MCP 模式同样生效（更优）；若不启动，则在 admin/混合模式补清理。无论哪种，都不会破坏 MCP 协议（Timer 回调走 stderr 日志，不碰 stdout）。
2. **配置热重载 + 后台服务读最新值**：后台服务每次 tick 都重新读 `ConfigStore.Current`，天然跟随热重载，无需额外通知机制。
3. **手动清理迁移不丢功能**：API 不变，仅前端位置变化，行为一致。
4. **天数校验**：与 `DeleteOlderThan` 现有约束（`days > 0`）一致；前端 + 后端双重校验。
5. **备份删除安全**：复用 `IsSafeBackupName` + 仅按文件时间判断，不误删非备份文件；恢复前快照机制不受影响（快照也是合法备份，按时间老化）。
6. **maintenance 节点向后兼容**：旧 config.json 无该节点时 null，自动清理默认关闭，无行为变化。
7. **只做最小改动**：不重构现有配置保存流程，不顺手改其它字段；settings 页独立保存，不与 projects/keywords 互相覆盖。

### 🔴 致命缺陷（必须修复，否则功能不可用或回归）

**D1：`PUT /admin/api/config` 全量替换会静默丢失 maintenance 节点**
- 证据：`AdminConfigService.ToConfig`（`src/McpDbTools.Server/Admin/AdminConfigService.cs:243-258`）重建 `DatabasesConfig` 时未复制 `Maintenance`；`record` 的 `init` 属性不赋值即默认（null）。
- 影响：用户在「项目配置」/「关键字」页保存后，之前在「全局设置」配的自动清理配置丢失，自动清理失效。
- 测试佐证：`AdminConfigServiceTests` 证实 `SaveConfigAsync` 走的就是 `ToConfig` 重建路径，无保留未知节点机制。
- **修复**：`ToConfig` 返回的 `new DatabasesConfig { ... }` 必须新增 `Maintenance = current.Maintenance`（从 current 透传）。并在 `AdminMaintenanceTests` 增加防回归测试：先保存 maintenance，再保存 projects，断言 maintenance 仍在 config.json。

**D2：`MaintenanceHostedService` 依赖 `AdminConfigService`，但后者只在 Admin 模式 DI 注册，MCP 模式注册会导致启动崩溃**
- 证据：`AdminConfigService` 仅在 `Program.cs:56`（`RunAdminAsync`）`AddSingleton`；`ConfigureBusinessServices`（`Program.cs:201-218`，MCP 模式也调用，见 `Program.cs:35`）里**没有**。
- 方案原文说"在 `ConfigureBusinessServices` 注册 `MaintenanceHostedService`" → 若它在 MCP host 启动，DI 解析 `AdminConfigService` 抛异常，MCP 进程启动失败。
- 注意：`AuditLogger`（`AuditLogger.cs:57-69`）从 `ConfigStoreOptions` 自己算 dbPath，不依赖 `AdminConfigService`，所以审计写入在 MCP 模式独立可用。
- **修复（二选一，需决策）**：
  - (a) 仅在 `RunAdminAsync` 注册 `MaintenanceHostedService`（与 `AdminConfigService` 同处），MCP 模式不自动清理。简单、零回归风险。审计清理在下次 Admin/混合模式运行时补做。
  - (b) 拆分依赖：`MaintenanceHostedService` 只依赖 `ConfigStore` + `AuditLogger` + 备份目录路径（从 `ConfigStoreOptions.ConfigPath` 算，与 `AdminConfigService` 同算法），把 `DeleteBackupsOlderThan` 的核心逻辑下沉到不依赖 `AdminConfigService` 的位置（如静态工具方法或 `MaintenanceHostedService` 内部），让 MCP 模式也能自动清理。
  - **推荐 (a)**：本工具是本机工具，Admin 模式常驻时清理即可；MCP 模式清理价值低且增复杂度。但需用户确认。

**D7：README/CLAUDE.md 明确承诺"不自动清理"，本次功能打破承诺但方案未列文档更新**
- 证据：`README.md:12`、`:363`：「不自动清理（用户不主动删除则一直保留）」；`README.md:151`、`:363`：「手动清理 30/60/90 天」（档位也变了，应改为 10/20/30/50）；`CLAUDE.md:86`：「固定保留 30 天并惰性清理」（此句本就与代码现状不符，需一并核对修正）。
- **修复**：改动文件清单新增 `README.md`（同步审计特性描述、手动清理档位、新增自动清理说明）与 `CLAUDE.md`（修正审计章节表述，描述 maintenance 节点）。

### 🟡 待完善（不阻塞但需明确，否则实现歧义或边界踩坑）

**D3：`SaveMaintenanceAsync` 实现路径含糊**
- 方案说"基于 current 深拷贝替换 maintenance"，但 `DatabasesConfig` 是 record、属性全 `init`，深拷贝需手写或 `with`。且 `WriteConfigAtomicallyAsync` 是 private，如何复用未说。
- **明确为**：`SaveMaintenanceAsync` 内部——① 取 `ConfigStore.Current`；② 用 `with { Maintenance = 解析后的新值 }` 产生新 `DatabasesConfig`（record with 表达式，浅拷贝足够，因其它字段引用不可变）；③ 调用与 `WriteConfigAtomicallyAsync` 相同的原子写流程。建议把 `WriteConfigAtomicallyAsync` 抽成 `private async Task<string> WriteAtomicallyAsync(DatabasesConfig config, ct)` 供 `SaveConfigAsync` 与 `SaveMaintenanceAsync` 共用（最小重构，仅提取已有方法，不改逻辑）。

**D5：天数边界/校验语义不完整**
- **明确为**：
  - 启用开关关闭（`*AutoCleanup=false`）时，对应天数不校验、不生效，可保留任意值（前端禁用编辑，后端忽略）。
  - 启用开关开启时，天数必须 `> 0`；前端校验 + 后端 `PUT /admin/api/maintenance` 校验，非法返回 400。
  - 后台服务 tick 时再次防御：开关开但天数 `<= 0` 则跳过该项清理并 LogWarning（不抛）。
  - `DeleteBackupsOlderThan(days)` 与 `DeleteOlderThan(days)` 一致：`days<=0` 抛 `ArgumentException`，由调用方（API 或后台服务）决定如何处理。

**D6：index.html 引入 settings.js 不应用 `defer`**
- 证据：现有 8 个脚本（`index.html:59-66`）均无 defer，靠加载顺序保证 `window.adminUi`/`window.adminViews` 先就绪。加 defer 会改变执行时序，与惯例冲突。
- **明确为**：去掉 defer，在 `index.html` 的 `<script src="./scripts/backups.js">` 之后、`shell.js` 之前插入 `<script src="./scripts/settings.js"></script>`。

**D8：自动删除备份与"恢复前快照"的语义需明确说明**
- `RestoreBackup`（`AdminConfigService.cs:393-428`）恢复时产生新快照（时间新）。自动删除只按 `LastWriteTimeUtc < cutoff` 判断，新生成的快照与保存备份天然不过期，安全。
- **明确为**：自动删除规则只看文件最后写入时间，不区分"恢复快照"与"保存备份"；恢复产生的新快照因时间新而安全。在方案风险节补充这句。

**D9：`MaintenanceConfig` 天数缺失/为 0 的回退**
- JSON 里 `maintenance` 存在但只写 `{"auditLogAutoCleanup": true}` 时，反序列化 `AuditLogRetentionDays=0`，`DeleteOlderThan(0)` 抛异常。
- **明确为**：`MaintenanceConfig` 提供单一默认常量真源：
  ```csharp
  public sealed class MaintenanceConfig
  {
      public const int DefaultRetentionDays = 30;
      public bool AuditLogAutoCleanup { get; init; }
      public int AuditLogRetentionDays { get; init; } = DefaultRetentionDays;
      public bool BackupAutoCleanup { get; init; }
      public int BackupRetentionDays { get; init; } = DefaultRetentionDays;
  }
  ```
  反序列化缺天数时取 30（record init 默认）。`GET /admin/api/maintenance` 当 `DatabasesConfig.Maintenance` 为 null 时，返回 `MaintenanceConfig` 内置默认（全 false + 30）。前端 settings.js 默认显示也用 30，与常量对齐。

**D10：1 小时周期 + 30 秒首启延迟依据，及重入保护**
